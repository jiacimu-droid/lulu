# 角色中立拟人状态设计

## 目标

保留现有 `LuluState`、`LuluPresence`、工具名和持久化字段以兼容已有用户数据，但清除确定性 fallback 中对露露、女性、管家、监督者和亲密肢体接触的强制假设。

角色的性格、关系位置、表达方式和职责必须来自 Assistant 的 system prompt、真实互动证据和统一 CompanionRuntime，而不是来自旧类名或固定中文模板。

## 规则

- 新角色的默认关系从谨慎开始，不默认“熟悉”或“很黏”。
- 用户沉默只表示对话暂停，不能自动推导为疏远、寂寞、想念或需要主动靠近。
- 确定性心声只描述观察、顾虑和选择，不擅自生成拥抱、贴贴、陪睡、坐在身边等身体行为。
- 表达素材提供强度、节奏和边界，不替角色决定可爱、温柔、恋爱或管家风格。
- 学习、截止时间和起床场景只在用户有明确目标或承诺时体现可靠度，不把所有角色固定成监督员或数字管家。
- 提示词使用“当前角色”或实际角色名，不使用固定“露露”或女性代词。
- 内部兼容标签如 `<lulu_presence>` 和工具名暂时保留，避免协议及历史消息迁移风险；它们不得出现在自然语言角色指令中。

## 范围

- 中立化 `LuluState` 默认值、沉默投影、内心 fallback 和自我场景。
- 中立化 `LuluPresence` 表达建议和具身素材。
- 中立化聊天规划、主动工具规划和滚动判断中的硬编码角色文案。
- 将状态上下文中的“亲密感”改为“关系位置”，将“靠近冲动”改为“未表达意图”。
- 更新对应纯 Kotlin 测试断言。

## 非目标

- 不重命名数据库字段、序列化类、Preferences key 或现有工具名。
- 不在本阶段删除旧状态 UI。
- 不强制所有角色冷淡；人设明确要求亲密、管家、监督或身体动作时，模型仍可自然表达。
- 不运行高成本 Gradle 编译；按用户要求采用静态检查并由用户做设备体验验证。
